# Contributing to AgriLink

Thank you for your interest in contributing to AgriLink! This document provides guidelines and instructions for contributing.

## Code of Conduct

- Be respectful and inclusive
- Welcome newcomers and help them learn
- Focus on constructive feedback
- Respect different viewpoints and experiences

## How to Contribute

### Reporting Bugs

1. Check if the bug has already been reported in the Issues section
2. If not, create a new issue with:
   - A clear, descriptive title
   - Steps to reproduce the bug
   - Expected vs actual behavior
   - Screenshots if applicable
   - Environment information (OS, device, app version)

### Suggesting Features

1. Check if the feature has already been suggested
2. Create a new issue with:
   - A clear description of the feature
   - Use cases and benefits
   - Any mockups or examples if applicable

### Pull Requests

1. Fork the repository
2. Create a feature branch (`git checkout -b feature/amazing-feature`)
3. Make your changes following the coding standards
4. Test your changes thoroughly
5. Commit your changes (`git commit -m 'Add amazing feature'`)
6. Push to the branch (`git push origin feature/amazing-feature`)
7. Open a Pull Request

## Development Setup

1. Follow the setup instructions in the main README.md
2. Ensure all tests pass before submitting
3. Run linting: `npm run lint`
4. Run type checking: `npm run type-check`

## Coding Standards

- Use TypeScript for all new code
- Follow React Native best practices
- Use functional components with hooks
- Keep components small and focused
- Add comments for complex logic
- Use meaningful variable and function names
- Follow the existing code style

## Commit Messages

- Use clear, descriptive commit messages
- Start with a verb (Add, Fix, Update, Remove, etc.)
- Keep the first line under 72 characters
- Add more details in the body if needed

Example:
```
Add plant disease diagnosis feature

- Integrate Plant.id API
- Add image picker component
- Display diagnosis results with confidence scores
```

## Testing

- Test on both Android and iOS if possible
- Test offline functionality
- Test with different screen sizes
- Verify error handling

## Questions?

Feel free to open an issue for any questions or clarifications.
